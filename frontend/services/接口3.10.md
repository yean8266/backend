#### **接口E：提交抽象论文 (Submit Paper)**

请求方式：POST /api/submit

前端发送的表单字段 (FormData)：

title (String)：论文标题（必填）

author (String)：作者/第一背锅侠（若用户未填，前端会默认传入 "佚名研究员"）

contact (String)：通讯作者联系方式（必填，用于发奖金。注意：此为敏感字段，必须入库，但绝对不能在 A 接口的列表里返回给前端）

abstract (String)：灾难复盘/摘要内容（必填）

link (String)：原文链接（选填）

file (File | Null)：用户上传的 PDF 文件（选填。前端已做 10MB 大小和 .pdf 格式拦截，后端建议再做一次安全校验）

link和file前端还要做一个逻辑二选一必须有一个



期望返回：{ "success": true }



算法思路：

1\.鉴权拦截：解析请求头中的 Token，验证用户是否已登录。若未登录，直接拒绝并返回 401 错误（不接收完全匿名的黑户提交）。

2\.文件上传处理：

检查请求包中是否包含 file 字段。

如果有文件，将其上传至云端对象存储（如 Supabase Storage 的特定 bucket 中）。

上传成功后，获取该文件的持久化下载链接（file\_url）。

3\.生成独立编号：为该篇论文生成一个唯一的 ID，建议采用格式：BNL-YYMM-随机码/自增序列（例如 BNL-2603-042），或者直接使用 UUID。

4\.数据入库：将文本数据（标题、作者、摘要、联系方式、外部链接）以及生成的 file\_url，一起 INSERT 插入到 papers 数据表中。

5\.初始化状态：

强制将新插入记录的 total\_votes 设为 0。

强制将新插入记录的 status 设为 'pending'（待审核状态）。注：如果你希望提交后立刻在前端大厅看到，这里可以设为 'approved'，具体视你们的运营策略而定。





#### **我觉得数据库里的数据属性：**

*斜体option* （括号里是参数属性）



**用户信息与状态表 (users 或 profiles)：**

id（UUID (主键)）

*nickname（Varchar）：默认null*

*avatar\_url（Varchar）：默认null，头像链接*

daily\_votes\_left（Integer）：默认5，今日剩余选票。必须加上 >= 0 的检查约束（Check Constraint），防止被扣成负数。

has\_shared\_today（Boolean）：默认false，今天是否领过分享奖励。领过后变 true。

last\_active\_date（Date/Timestamp）：默认now（），防刷与跨天重置基准。每次用户发起请求时，后端检查此字段：如果不是今天，就把票数重置为 5，分享状态重置为 false，并更新此字段为今天。

created\_at（Date/Timestamp）：默认now（），账号创建时间。



**抽象论文/项目表 (papers)：存储通过 /submit 接口提交上来的所有数据，并缓存总票数以供排行榜快速查询**

id（Varchar/UUID (主键)）：论文的唯一编号（如 BNL-2603-001）或默认 UUID。

title（Varchar）：论文标题。

author（Varchar）：默认'佚名研究员'。

**contact（Varchar）：通讯作者联系方式（微信号等）。注意：后端在写 GET /api/papers 时，绝对不能把这个字段查出来发给前端，仅限管理员后台看。在网络传参的时候，不要再F11里被看到。**

abstract（Text）：摘要

evidence\_link（Varchar）：原文链接。

file\_url（Varchar）：新文件上传，指向 Supabase Storage 里的 PDF 文件下载链接。

total\_votes（Integer）：默认0，该文章的总得票数。用于排行榜 sort=hot 排序。

status（Varchar）：默认'pending'，状态控制：pending (待审核), approved (已过审，可在前端展示), rejected (打回)。

created\_at	（Timestamp）：默认now()，提交时间。用于排行榜 sort=new 最新入库的排序。
